2.2
Enable Firestore Database
Inside your Firebase project, go to Build → Firestore Database → Create Database. When it asks for a mode, choose Start in production mode. Pick the server location closest to you (e.g. us-central1 for US).
Firebase Console → Firestore Firestore Quickstart Docs
Why production mode? It locks the database down by default. You control exactly who can read/write — in this case, only requests that include a valid Whop license key.
2.3
Set Firestore security rules
In Firestore go to the Rules tab and paste these rules. They allow any request that includes a valid user ID (the Whop license key) to read and write only their own data — nobody else's.
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId}/{document=**} {
      allow read, write: if request.auth == null
        && userId == request.resource.data.licenseKey;
    }
  }
}
Note: Come back here after creating the project and I'll set the exact rules that match the extension's data structure.
2.4
Get your Firebase config keys
Go to Project Settings → General → Your Apps → Add App → Web. Give the app a nickname (e.g. "Card Notes Pro Extension") and click Register. Firebase will show you a config object like this — copy the whole thing:

5.2
How sync works in practice
Every write (new note, rating, tag) goes to both chrome.storage locally and Firestore in the cloud. On a new device, entering the license key triggers a pull from Firestore — all their notes appear immediately. If they're offline, writes queue locally and sync the moment they reconnect.
